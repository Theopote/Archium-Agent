# v0.2 Alpha Validation Sprint

> **目标不是增加功能，而是证明现有系统可用。**

Stage 1–16 已完成工程与主链搭建。本冲刺**不新增 Stage**，改用真实建筑项目验收输出质量。

## 核心问题

Archium 能否把一个**真实、混乱、包含矛盾资料**的建筑项目，稳定变成一份建筑师**愿意继续编辑并拿去汇报**的 PPT？

在此问题通过真实案例验证之前，Archium 仍是**架构优秀的 Alpha**，而不是成熟的建筑汇报产品。

## 完整主链（已实现）

```
资料 → 事实 → Brief → Storyline → SlideSpec
     → 引用 → 素材 → 四层审核 → 修复 → JSON / PPTX / PDF
```

本冲刺验证的是**整条链在真实输入下的稳定性与输出质量**，而非继续扩展节点数量。

## 必须完成

| 工作项 | 说明 |
|--------|------|
| CI 稳定绿 | GitHub Actions 在默认分支 `master` 上 3.11 / 3.12 持续通过 |
| 三个 Golden Case | 医院老院区 / 校园改造 / 概念投标，固定资料 + 预期结果 |
| 导出产物归档 | 保存 PresentationSpec、JSON；可选 PPTX / PDF / 预览图 |
| 页面级视觉回归 | 对预览图或渲染截图建立 baseline 对比 |
| Checkpointer 生命周期 | SQLite 连接可关闭，Windows 下可删除临时库 |
| 修复后复审 | 自动修复 → 清空旧 issue → 四层复审（有轮次上限） |
| ReviewIssue 不重复 | 去除 reducer 导致的 issue 指数累积 |
| 事实冲突与单位 | 语义别名冲突、单位标准化、人工确认路径 |
| 跨平台渲染 | 图片路径、中文字体、Node.js PptxGenJS 在 Win/macOS/Linux |
| 用户修改成本 | 记录真实用户编辑行为，评估返工时间 |

## 完成标准

本冲刺**完成**当且仅当：

1. 验收清单中**基础设施**与**输出质量**项全部勾选；
2. 三个 Golden Case 在 CI 上连续绿至少一周；
3. 至少一名非开发用户在真实项目上完成「导入 → 生成 → 编辑 → 导出」并记录修改成本。

完成后进入 **v0.2-beta** 规划，而非 Stage 17。

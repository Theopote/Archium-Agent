## 当前实现分析

### 文件：`deck_composition_service.py`

#### ✅ 优点

**1. 清晰的架构**
- 分离关注点：`plan()` 初始规划，`revise()` 修订
- 模块化的节奏调整方法
- 良好的数据结构（`DeckCompositionPlan`, `SlideCompositionDirective`）

**2. 基础启发式规则**
- ✅ 版式多样性：避免连续 3 页相同 LayoutFamily
- ✅ 密度节奏：高密度后安排低密度缓冲
- ✅ 文字节奏：避免连续 3 页纯文字
- ✅ 图纸节奏：避免连续 3 页图纸

**3. 锚点识别**
- 识别 hero 页面、过渡页面、高潮页面
- 章节策略构建

#### ❌ 限制和问题

### 1. `revise()` 方法过于简单

**当前实现**（第 133-182 行）：
```python
def revise(self, plan, feedback: str, ...):
    normalized = feedback.strip().lower()
    
    # 仅基于关键词匹配
    if any(token in normalized for token in ("节奏", "pacing", "单调")):
        # 简单设置 should_contrast_previous = True
        
    if any(token in normalized for token in ("视觉", "hero", "主图", "图纸")):
        # 简单增加 hero_priority + 0.15
        
    if any(token in normalized for token in ("文字", "text", "密度", "density")):
        # 简单减少 text_priority - 0.1
```

**问题**：
- ❌ 仅依赖关键词匹配，无语义理解
- ❌ 修改幅度固定（+0.15, -0.1），不考虑实际情况
- ❌ 不考虑页面上下文关系
- ❌ 不考虑用户历史调整模式

### 2. 缺乏多维度信息整合

**当前未使用的信息源**：

#### A. Deck QA 反馈
- ❌ 不读取 `DeckQAReport` 的具体问题
- ❌ 不针对具体问题（footer 不一致、版式重复）做针对性修复
- ❌ 不使用 QA 的严重程度指导调整力度

#### B. 章节语义
- ❌ 仅用 `chapter_id` 分组，不分析章节主题
- ❌ 不考虑章节间的逻辑关系（递进、对比、并列）
- ❌ 不根据章节重要性调整视觉强度

#### C. 页面截图
- ❌ 不分析实际渲染效果
- ❌ 不检测视觉平衡问题（元素过挤、留白过多）
- ❌ 不评估实际的视觉强度（仅用预设值）

#### D. 前后页视觉强度
- ❌ 仅检查密度级别，不综合评估视觉冲击力
- ❌ 不计算视觉强度梯度（突变 vs 渐变）
- ❌ 不考虑整体节奏曲线的平滑性

#### E. 用户调整历史
- ❌ 不记录用户的修改模式
- ❌ 不学习用户偏好
- ❌ 每次 `revise()` 都是独立的，没有累积学习

### 3. 规则过于简单和固定

**硬编码的规则**（第 39-49 行）：
```python
_DEFAULT_VARIETY_RULES = (
    "连续三页不得使用相同 LayoutFamily",  # 固定阈值
    "连续两页不得使用完全相同 Variant",
    "每个主要章节至少有一个视觉强页",
    # ...
)
```

**问题**：
- ❌ 阈值固定（3 页），不适应不同长度的 Deck
- ❌ 规则互相冲突时无优先级
- ❌ 不考虑特殊场景（技术文档 vs 营销材料）

### 4. 缺乏反馈循环

**当前流程**：
```
plan() → 生成初始计划
   ↓
revise(feedback) → 关键词匹配 → 微调参数
   ↓
生成新计划
```

**缺失**：
- ❌ 不验证修改是否解决了问题
- ❌ 不对比修改前后的 QA 分数
- ❌ 不记录哪些调整有效

### 5. 优先级计算过于简单

**`_content_priorities()` 函数**（第 433-450 行）：
```python
def _content_priorities(intent: VisualIntent) -> tuple[float, float, float]:
    content = intent.dominant_content_type
    if content == VisualContentType.SITE_PLAN:
        return 0.35, 0.25, 0.9  # 硬编码值
    # ...
```

**问题**：
- ❌ 固定的优先级值，不考虑内容复杂度
- ❌ 不考虑素材质量（模糊图 vs 高清图）
- ❌ 不考虑用户偏好

## 具体场景分析

### 场景 1：用户反馈"节奏太单调"

**当前处理**：
```python
if "节奏" in feedback:
    for directive in revised.slide_directives:
        directive.should_contrast_previous = True
```

**问题**：
- 所有页面都设置对比，可能过度矫正
- 不分析哪些页面真的单调
- 不考虑章节边界

**应该怎么做**：
1. 分析视觉强度曲线，找出平坦区域
2. 识别连续相似的页面
3. 针对性插入对比页面
4. 验证修改后的节奏曲线

### 场景 2：用户反馈"主图不够突出"

**当前处理**：
```python
if "主图" in feedback:
    for directive in revised.slide_directives:
        if directive.hero_priority >= 0.5:
            directive.hero_priority += 0.15  # 固定增量
```

**问题**：
- 所有 hero 页都增加优先级，可能不必要
- 不分析哪些页面的主图真的不突出
- 不考虑图片质量和内容重要性

**应该怎么做**：
1. 读取页面截图，分析主图实际占比
2. 评估主图视觉吸引力
3. 识别关键页面（标题、章节开头、高潮）
4. 针对性增强这些页面的主图
5. 验证修改后的视觉层次

### 场景 3：DeckQA 报告多个 footer 不一致

**当前处理**：
- ❌ 完全不处理 DeckQA 的具体问题

**应该怎么做**：
1. 读取 DeckQAReport
2. 识别 footer 位置不一致的页面
3. 找出主流位置
4. 调整异常页面的 chrome 布局
5. 重新运行 QA 验证

## 改进方向总结

### 短期改进（1-2 周）

1. **增强 `revise()` 语义理解**
   - 使用 LLM 解析反馈意图
   - 提取具体的问题页面和期望调整

2. **集成 DeckQA 反馈**
   - 读取 QA 报告的具体问题
   - 针对性修复高严重度问题

3. **视觉强度分析**
   - 计算实际的视觉强度曲线
   - 识别单调区域和突变点

### 中期改进（1-3 月）

4. **章节语义分析**
   - 使用 LLM 分析章节主题和关系
   - 根据章节重要性调整视觉策略

5. **页面截图分析**
   - 提取视觉特征（颜色、对比度、元素密度）
   - 评估实际视觉效果

6. **用户历史学习**
   - 记录用户修改模式
   - 学习用户偏好
   - 预测用户可能的调整

### 长期改进（3-6 月）

7. **智能规则系统**
   - 动态调整规则阈值
   - 规则冲突解决
   - 场景自适应规则

8. **端到端优化**
   - 直接从反馈生成最优方案
   - 多目标优化（节奏、一致性、视觉吸引力）

9. **交互式调整**
   - 实时预览调整效果
   - A/B 对比方案
   - 迭代优化

## 技术债务

1. **硬编码值过多** - 需要配置化或学习
2. **缺乏测试** - 需要增加单元测试和集成测试
3. **缺乏日志** - 难以调试和优化
4. **性能问题** - 每次 revise 重新 plan，效率低

## 结论

当前的 `DeckCompositionPlanningService` 是一个良好的起点，但在以下方面有显著限制：

1. **过度依赖关键词匹配** - 需要语义理解
2. **信息孤岛** - 需要整合多维度信息
3. **固定规则** - 需要自适应和学习
4. **缺乏验证** - 需要闭环反馈

下一步应该创建一个增强版本，逐步整合这些改进点。
